Problem Statement

  • ESCO taxonomy defines 13,890 standardized skill and knowledge labels
  • Goal: given a skill mention, retrieve the correct ESCO entry

Why create our own benchmark datasets?

Opaque annotation quality

The quality of existing gold labels is not fully transparent — little detail on annotators, annotation protocols, or inter-coder reliability.

Sectoral bias

Current benchmarks are skewed toward tech and service-sector vacancies, limiting representativeness for other occupations and industries.

No Polish benchmark

No widely used Polish benchmark datasets currently exist, creating a major evaluation gap for Polish labour market data.

Why Synthetic Data for ESCO Embeddings?

Scale & coverage

Manually annotated skill datasets are scarce and expensive — synthetic data generates large-scale training pairs covering much more of the taxonomy

Realistic language

Synthetic examples resemble real job-ad language, so the model learns to recognize skills expressed indirectly rather than named explicitly

🔄Diverse realizations via LLMs

LLMs generate paraphrases, implicit mentions, and job-ad style sentences beyond simple dictionary matching

JobSkape generates realistic synthetic postings for skill matching

⛏️Hard negatives matter

Strong retrieval models rely on synthetic data generation combined with hard-negative mining

Central to NV-Embed and domain-adapted ESCOXLM-R

Challenges in Synthetic Data Quality

1
Lexical leakage

LLM copies the ESCO label verbatim — model learns surface overlap, not semantics

explicitly address this with more diverse generation

2
Unrealistic positives

Short definitional sentences are too easy and don't resemble real job-ad language

JobSkape motivation: earlier synthetic examples lacked recruitment-style phrasing

3
Weak negatives

Random negatives are trivial; training needs sibling skills and lexically similar distractors

Hard-negative design is central in and

4
Ambiguity

A span may map to different ESCO skills depending on occupational or sentential context

Taxonomy-aware models like ESCOXLM-R partially address structural cues

5
Evaluation mismatch

RP@1 alone is insufficient — MRR, RP@k, and ranking pipelines provide a fairer picture

Practical implication

The strongest setup combines contrastive training with diverse positives, realistic phrasing, and hard negatives — optionally followed by a two-stage bi-encoder → cross-encoder pipeline

Synthetic Training Data & Fine-Tuning

Synthetic training data
  • Generated using ESCO skill labels and descriptions
  • Prompted Claude Haiku to produce realistic sentences without altering original skill labels
  • ~10 examples per skill → ~140k sentence–skill pairs
Model training
  • Contrastive learning — pull matching sentence–skill pairs closer, push non-matching apart
  • Fine-tuned two models:
voyage-4-nano · ~400M params NV-Embed-v2 · ~7.85B params (Mistral-7B)
Parameter-efficient fine-tuning
  • Applied LoRA (Low-Rank Adaptation) — learns small low-rank adapters added to existing layers
  • No need to update all model weights
  • Much cheaper training while still adapting the embedding space to the skill-matching task

Bi-encoder & Cross-encoder for Skill Matching

Bi-encoder — retrieval stage
  • Encodes job sentence/span and ESCO skill independently into a shared embedding space
  • Similarity computed with cosine distance
  • Enables scalable retrieval from large taxonomies (~13k skills)
  • Trained with contrastive learning — positive: sentence–correct skill; negatives: other skills in batch
  • Outputs top-K candidate skills for reranking

· ·

Cross-encoder — reranking stage
  • Evaluates sentence + candidate skill jointly in a single forward pass
  • Input: [CLS] sentence [SEP] skill description
  • Transformer predicts relevance score for each candidate
  • Captures fine-grained contextual interactions → improves ranking accuracy
  • Applied only to top-K candidates — too expensive for full taxonomy

Used in our benchmark as second-stage reranker (rerank-2.5) · ·

Evaluation Metrics for Skill Retrieval

RP@K (R-Precision at K)

Measures how many correct labels appear among the top-K ranked predictions.

  • The model produces a ranked list of candidate skills for each sentence
  • RP@K counts how many gold labels appear in the first K results
  • Score normalized by \(\min(K, R_n)\), then macro-averaged
\[\text{RP@K} = \frac{1}{N}\sum_{n=1}^{N}\sum_{k=1}^{K}\frac{\text{Rel}(n,k)}{\min(K,\,R_n)}\]

\(\text{Rel}(n,k)\) is a binary indicator of whether the \(k\)-th ranked label is correct for sample \(n\), and \(R_n\) is the number of gold labels for sample \(n\).

MRR (Mean Reciprocal Rank)

Measures how high the first correct label appears in the ranking.

  • For each example, compute \(1/\text{rank}_i\) of the first correct prediction
  • Final score is the average across all examples
  • MRR = 1.0 means the correct skill is always ranked first
\[\text{MRR} = \frac{1}{N}\sum_{i=1}^{N}\frac{1}{\text{rank}_i}\]

\(\text{rank}_i\) is the position of the first correct ESCO label in the ranked list for sample \(i\).

Approach

1. Embed ESCO Labels

Encode all 13,890 preferredLabel entries into dense vectors using an embedding model. Store in a FAISS index for fast retrieval.

2. Embed Query Spans

Encode each extracted skill span / sub_span from test annotations using the same model.

3. Nearest-Neighbour Search

For each query, retrieve top-K nearest ESCO labels by cosine similarity using FAISS IndexFlatIP.

4. Evaluate vs Gold Labels

Compare predicted ESCO labels against human-annotated gold_label. Compute RP@K and MRR.

Embedding Strategies

Label-only

  • ESCO: embed preferredLabel
  • Query: embed span / sub_span
  • Simple, fast, best performing in most cases

Contextual

  • ESCO: embed label [SEP] description
  • Query: embed span [SEP] sentence
  • Adds context but dilutes the skill signal
  • Only VoyageAI context-3 handles this well natively

Simple concatenation with [SEP] degraded performance for OpenAI and ESCOXLM-R. Native contextual models (voyage-context-3) fared better.

Two-Stage Retrieval: FAISS + Reranker

  • Stage 1: Bi-encoder retrieves top-100 candidates from FAISS
  • Stage 2: Cross-encoder (rerank-2.5) re-scores each (query, candidate) pair jointly
  • Cross-encoders are more expressive but cannot pre-compute embeddings
Result: the general-purpose Voyage reranker degraded performance compared to domain-specific bi-encoders. Skill matching benefits from specialization over raw model capacity.
